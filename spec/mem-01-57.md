# AD-mlnf-mem 记忆中枢 · 模块总览

> 所属仓库: AD-mlnf-mem
> 对应架构: AD-Core 自动驾驶认知系统 V1.0
> 底层理论: MLNF-Mem 多级嵌套漏斗记忆系统（原创提出者：文波福，2026年3月17日全球首发）
> 模块总数: 67个（57复用HR + 3适配 + 7新增）


## 一、模块来源说明

AD-mlnf-mem 的67个模块中，**01-57号模块完全复用 HR-mlnf-mem 的成熟设计**。这57个模块是 MLNF-Mem 的“记忆操作系统”，管的是经验的晋升、遗忘、重要度计算、宏观自收敛——这些规则与记忆内容无关，驾驶场景与机器人场景完全通用。

**为什么通用？**

- mem-26（记忆晋升双条件判定）：只管“时间达标+重要度达标→晋升”，不关心经验内容是“抓杯子”还是“变道”
- mem-28（遗忘阈值判定）：只管“重要度太低+复用不足→遗忘”，不关心被删的是什么
- mem-44（条件化记忆检索查询）：只管“根据条件检索记忆”，不关心检索出来的是什么内容

**这就像MySQL可以存淘宝订单也可以存医院病历——存储引擎通用，上层数据不同。**

## 三、给AD工程师的HR复用指引

AD工程师在开发AD-mlnf-mem时，**无需重新设计或重新编写01-57号模块**。这些模块的实现代码、接口规格、单元测试，全部直接引用HR-mlnf-mem仓库的对应文件。

**操作步骤：**

1. 访问 HR-mlnf-mem 仓库的 `spec/` 目录，获取 mem-01 至 mem-57 的完整接口规格
2. 访问 HR-mlnf-mem 仓库的 `src/` 目录，获取 mem-01 至 mem-57 的代码骨架
3. 这些模块的实现逻辑不做任何修改，直接集成到AD-mlnf-mem工程中
4. AD场景特有的记忆内容（驾驶经验、交通规则、神态分析记录等）由上层数据配置决定，不影响模块本身

## 四、模块编号锁定规则

1. mem-01 ~ mem-57：漏斗内核集群，**永久锁定**，版本内禁止新增、删减、重排
2. mem-58 ~ mem-60：外置扩展区，保留编号，仅做AD场景适配，不改变核心职能
3. mem-61 ~ mem-67：AD方向新增模块，编号从61开始向后追加，不得修改已有编号
4. 本地不再新增记忆类独立模块，高阶拓展能力统一由云端技能包迭代补充
