# ad-49-存储压缩与冷归档单元 接口规格

---

## 基本信息

| 项 | 内容 |
|----|------|
| 模块编号 | ad-49 |
| 模块名称 | 存储压缩与冷归档单元 |
| 所属分区 | 五、存储与系统运维 |
| 核心职责 | 对 L3 及以上层级被遗忘的经验条目执行压缩归档，将已压缩的经验数据转存至冷存储分区。接收来自 ad-42 的归档数据，存储于独立于活跃漏斗存储的冷归档分区，并维护归档索引以供后续追溯恢复。释放主存储空间的同时确保经验数据可恢复。在压缩、存储与恢复全过程中校验数据完整性 |
| 依赖模块 | ad-42（冗余记忆删除与归档单元，发送归档数据包）、ad-47（疑问缓存库，接收法规相关条目的永久归档） |
| 被依赖模块 | ad-42（接收归档写入确认）、ad-24/26（L3/L4 存储单元，接收归档恢复请求） |


## 内部状态定义

| 状态 | 标识 | 含义 | 触发条件 |
|------|------|------|----------|
| 正常运行 | `NORMAL` | 正常接收归档数据并压缩存储 | 系统初始化完成，冷归档分区挂载校验通过 |
| 压缩存储中 | `STORING` | 正在执行数据压缩与冷存储写入 | 收到归档数据包 |
| 归档恢复中 | `RESTORING` | 正在从冷归档分区恢复指定条目至原层级 | 收到合法的恢复请求 |
| 归档清理中 | `CLEANING` | 正在按保留策略清理过期归档条目 | 归档区容量超限或定期清理周期触发 |
| 存储降级 | `DEGRADED` | 冷归档分区可用容量不足或硬件异常，暂停新归档 | 冷归档分区使用率 > 95% 或硬件告警 |
| 暂停服务 | `PAUSED` | 系统紧急熔断或漏斗二冻结 | 收到全局冻结或紧急熔断指令 |


## 输入数据

| 输入项 | 数据类型 | 来源模块 | 触发条件 | 优先级 |
|--------|----------|----------|----------|:---:|
| 归档数据包 | Struct（条目 ID + 压缩经验数据 + 元数据 + 归档原因 + 原层级 + 原分槽号 + 归档时间戳） | ad-42 冗余记忆删除与归档单元 | 遗忘流程中冷归档条目执行时 | **高** |
| 法规模糊地带归档请求 | Struct（疑问条目 ID + 完整场景数据 + 归档原因 + 提交时间戳） | ad-47 疑问缓存库 | 法规模糊地带疑问到期且标记为永久归档 | 普通 |
| 归档恢复请求 | Struct（条目 ID + 原层级 + 恢复原因 + 请求来源模块） | ad-24/26 L3/L4 存储单元、ad-03 | 需要回溯查阅已归档经验时 | 普通 |
| 定期清理触发信号 | Signal | 系统时钟 | 每 30 日或冷归档分区使用率 > 80% | 普通 |
| 冷归档分区健康报告 | Struct（总容量 + 已用容量 + 坏块数 + 读写错误率） | 存储介质自检模块 | 每 60 秒周期性上报 | 普通 |
| 全局调度指令 | Enum（冻结/激活） | ad-03 漏斗二专属调度单元 | 模式切换时 | **高** |
| 紧急熔断指令 | Enum（全局熔断） | ad-01 总控漏斗 F₀ | 安全急停或碰撞事件时 | **紧急** |


## 输出数据

| 输出项 | 数据类型 | 目标模块 | 输出条件 | 优先级 |
|--------|----------|----------|----------|:---:|
| 归档写入确认 | Struct（条目 ID + 归档结果 + 归档位置索引 + 压缩前后大小 + 校验和 + 时间戳） | ad-42 | 归档写入完成后 | **高** |
| 法规模糊归档确认 | Struct（疑问条目 ID + 归档结果 + 归档位置索引 + 时间戳） | ad-47 | 法规模糊归档完成后 | 普通 |
| 归档恢复数据包 | Struct（条目 ID + 解压后完整经验数据 + 元数据 + 恢复时间戳） | 请求来源模块 | 恢复操作完成后 | **高** |
| 恢复失败通知 | Struct（条目 ID + 失败原因 + 建议操作） | 请求来源模块 | 归档数据损坏或条目不存在 | **高** |
| 归档清理汇总报告 | Struct（清理时间 + 清理条目数 + 释放空间量 + 剩余归档条目数 + 清理标准） | ad-03、ad-01、ad-51 | 定期清理完成后 | 普通 |
| 容量告警 | Struct（当前使用率 + 剩余容量 + 预估可支撑天数 + 建议操作） | ad-03、ad-01 | 冷归档分区使用率 > 80% 时 | **高** |
| 运行状态上报 | Struct（当前状态 + 归档条目总数 + 总占用空间 + 压缩率统计 + 恢复成功率） | ad-03、ad-01 | 每 60 秒周期性上报 | 普通 |


## 核心处理逻辑

```
FUNCTION cold_archive_main_loop():
    SET internal_state = NORMAL
    冷归档分区路径 = 独立存储分区（与活跃漏斗存储物理隔离）
    归档索引表 = {}  // 条目 ID → {归档位置, 压缩大小, 原大小, 校验和, 归档时间, 原层级, 原分槽, 归档原因}
    压缩算法 = LZ4（默认，可在 Zstandard/LZ4 之间切换）
    归档保留期限 = 1 年（默认，可配置）
    最大归档容量 = 50MB（默认，可配置）

    WHILE 系统运行中:
        // 第 1 步：最高优先级——紧急熔断
        IF 收到紧急熔断指令:
            SET internal_state = PAUSED
            CONTINUE
        ELSE IF internal_state == PAUSED AND 紧急解除:
            SET internal_state = NORMAL

        // 第 2 步：接收冷归档分区健康报告
        IF 收到冷归档分区健康报告:
            IF 健康报告.坏块数 > 安全阈值 OR 健康报告.读写错误率 > 安全阈值:
                SET internal_state = DEGRADED
                向 ad-03、ad-01 发送容量告警(原因=硬件异常)
                CONTINUE
            ELSE IF 健康报告.已用容量 / 健康报告.总容量 > 0.80:
                向 ad-03、ad-01 发送容量告警(原因=使用率>80%)
            ELSE IF 健康报告.已用容量 / 健康报告.总容量 > 0.95:
                SET internal_state = DEGRADED
                向 ad-03、ad-01 发送容量告警(原因=使用率>95%，暂停新归档)
                CONTINUE
            ELSE IF internal_state == DEGRADED AND 健康报告.已用容量 / 健康报告.总容量 < 0.70:
                SET internal_state = NORMAL

        // 第 3 步：接收并处理归档数据包
        IF 收到 ad-42 归档数据包 AND internal_state IN [NORMAL, STORING]:
            IF internal_state == DEGRADED:
                向 ad-42 返回归档写入确认(失败, 原因=冷归档分区不可用)
                CONTINUE

            SET internal_state = STORING
            条目 ID = 数据包.条目 ID
            原大小 = SIZE(数据包.压缩经验数据)
            元数据 = 数据包.元数据

            // 3a. 进一步压缩（如原数据未充分压缩）
            压缩后数据 = 压缩算法(数据包.压缩经验数据)
            压缩后大小 = SIZE(压缩后数据)
            校验和 = SHA256(压缩后数据)

            // 3b. 写入冷归档分区
            归档位置 = 分配冷存储空间(压缩后大小)
            IF 归档位置 == NULL:
                向 ad-42 返回归档写入确认(失败, 原因=冷归档空间不足)
                SET internal_state = NORMAL
                CONTINUE

            写入结果 = 写入冷归档分区(归档位置, 压缩后数据)
            IF 写入结果 != 成功:
                向 ad-42 返回归档写入确认(失败, 原因=存储写入失败)
                SET internal_state = NORMAL
                CONTINUE

            // 3c. 更新归档索引表
            归档索引表[条目 ID] = {
                归档位置: 归档位置,
                压缩后大小: 压缩后大小,
                原大小: 原大小,
                校验和: 校验和,
                归档时间: NOW(),
                原层级: 数据包.原层级,
                原分槽号: 数据包.原分槽号,
                归档原因: 数据包.归档原因,
                压缩算法: 压缩算法,
                元数据摘要: 提取摘要(元数据)
            }

            // 3d. 持久化归档索引
            保存归档索引表至冷归档分区

            // 3e. 返回确认
            向 ad-42 返回归档写入确认(条目 ID, 成功, 归档位置, 压缩后大小, 原大小, 校验和)

            // 3f. 记录归档日志
            向 ad-51 写入冷归档日志(条目 ID + 归档位置 + 压缩率 + 时间戳)

            SET internal_state = NORMAL

        // 第 4 步：处理法规模糊地带归档
        IF 收到 ad-47 法规模糊地带归档请求 AND internal_state IN [NORMAL, STORING]:
            SET internal_state = STORING
            疑问条目 ID = 请求.疑问条目 ID
            // 压缩并写入冷归档分区（流程同第 3 步，归档原因=法规模糊地带）
            ...
            向 ad-47 返回法规模糊归档确认(成功)
            SET internal_state = NORMAL

        // 第 5 步：处理归档恢复请求
        IF 收到归档恢复请求 AND internal_state IN [NORMAL, DEGRADED]:
            SET internal_state = RESTORING
            条目 ID = 请求.条目 ID
            请求来源 = 请求.请求来源模块

            // 5a. 检查归档是否存在
            IF 条目 ID NOT IN 归档索引表:
                向请求方返回恢复失败通知(条目 ID, "条目不在归档库中")
                SET internal_state = NORMAL
                CONTINUE

            归档信息 = 归档索引表[条目 ID]

            // 5b. 读取压缩数据
            压缩数据 = 读取冷归档分区(归档信息.归档位置, 归档信息.压缩后大小)
            IF 压缩数据 == NULL:
                向请求方返回恢复失败通知(条目 ID, "归档数据读取失败")
                SET internal_state = NORMAL
                CONTINUE

            // 5c. 校验数据完整性
            当前校验和 = SHA256(压缩数据)
            IF 当前校验和 != 归档信息.校验和:
                向请求方返回恢复失败通知(条目 ID, "校验和不匹配，数据可能已损坏")
                向 ad-03、ad-01 上报告警
                SET internal_state = NORMAL
                CONTINUE

            // 5d. 解压数据
            解压后数据 = 解压算法(压缩数据)
            IF 解压失败:
                向请求方返回恢复失败通知(条目 ID, "解压失败")
                SET internal_state = NORMAL
                CONTINUE

            // 5e. 返回恢复数据
            恢复数据包 = {
                条目 ID: 条目 ID,
                解压后完整经验数据: 解压后数据,
                元数据: 归档信息.元数据摘要,
                恢复时间戳: NOW()
            }
            向请求方返回归档恢复数据包(恢复数据包)
            向 ad-51 写入归档恢复日志(条目 ID + 恢复时间戳)

            SET internal_state = NORMAL

        // 第 6 步：定期清理过期归档
        IF 收到定期清理触发信号 OR (归档分区使用率 > 80% AND 距上次清理 >= 7 日):
            SET internal_state = CLEANING
            清理统计 = {清理条目数: 0, 释放空间: 0}
            当前时间 = NOW()

            FOR EACH 条目 ID, 归档信息 IN 归档索引表:
                应清理 = False
                归档时长 = 当前时间 - 归档信息.归档时间

                // 6a. 法规模糊地带：永久保留，不清理
                IF 归档信息.归档原因 == "法规模糊地带":
                    CONTINUE

                // 6b. 不可抗力标记：永久保留
                IF 归档信息.元数据摘要.不可抗力标记 == True:
                    CONTINUE

                // 6c. 超期清理
                IF 归档时长 > 归档保留期限:
                    应清理 = True
                // 6d. 容量压力清理（从旧到新）
                ELSE IF 归档分区使用率 > 90% AND 归档时长 > 归档保留期限 × 0.5:
                    应清理 = True

                IF 应清理:
                    安全覆写冷归档分区(归档信息.归档位置, 归档信息.压缩后大小)
                    REMOVE 条目 ID FROM 归档索引表
                    清理统计.清理条目数++
                    清理统计.释放空间 += 归档信息.压缩后大小

            // 保存更新后的归档索引表
            保存归档索引表至冷归档分区

            // 生成清理报告
            向 ad-03、ad-01、ad-51 发送归档清理汇总报告(清理统计)

            // 若清理后仍超 80%，上报告警
            IF 归档分区使用率 > 80%:
                向 ad-03、ad-01 发送容量告警(清理后仍超阈值)

            SET internal_state = NORMAL

        // 第 7 步：周期性状态上报
        IF 距上次上报 >= 60 秒:
            计算压缩率统计（平均压缩率 = 1 - 总压缩大小/总原大小）
            向 ad-03、ad-01 上报运行状态

        SLEEP 200ms
```


## 约束与异常处理

| 场景 | 处理方式 | 恢复条件 |
|------|----------|----------|
| 冷归档分区写入失败（坏块） | 尝试备用存储块，若均失败则标记该分区为 DEGRADED，上报告警 | 硬件修复或更换存储介质 |
| 归档数据校验和不匹配 | 该条目标记为“已损坏”，从索引表中移除，上报告警 | — |
| 归档恢复时解压失败 | 标记该条目标记为“可能损坏”，上报告警，建议从备份恢复 | 备份恢复或手动修复 |
| 冷归档分区使用率 > 95% | 状态=DEGRADED，暂停新归档，触发紧急清理（从最旧条目开始） | 使用率回落至 < 70% |
| 法规模糊地带归档条目尝试恢复 | 正常支持恢复，但记录审计日志 | — |
| 同一归档条目在短时间内多次请求恢复 | 缓存解压结果 30 秒，避免重复解压 | — |
| 归档索引表损坏 | 从冷归档分区头部扫描重建索引，上报告警 | 重建完成 |
| 压缩算法切换 | 新归档使用新算法，已有归档保持原算法不解压重压缩 | — |
| 紧急熔断期间收到归档请求 | 暂存于 ad-42，紧急解除后补发 | 紧急解除 |

## 安全边界

| 规则编号 | 内容 |
|:---:|------|
| S-01 | 冷归档分区与活跃漏斗存储分区物理隔离，运行时仅本模块拥有冷归档分区的写入权限，其他模块只读 |
| S-02 | 归档数据写入后即执行 SHA256 校验和计算，恢复时必须通过校验和验证方可解压返回，确保数据零损坏 |
| S-03 | 法规模糊地带归档条目享有永久保留权限，任何清理操作不得删除此类条目 |
| S-04 | 不可抗力标记的经验条目在归档后仍享有永久保留权限，清理操作跳过 |
| S-05 | 过期归档条目的删除须执行 DoD 5220.22-M 标准单次覆写，不可通过文件系统删除直接操作 |
| S-06 | 归档索引表每完成一次写入或清理操作后自动持久化至冷归档分区，确保断电不丢失索引 |
| S-07 | 所有归档写入、恢复、清理操作全量写入 ad-51 不可变日志 |
