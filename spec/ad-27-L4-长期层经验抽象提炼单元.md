# ad-27-L4-长期层经验抽象提炼单元 接口规格

---

## 基本信息

| 项 | 内容 |
|----|------|
| 模块编号 | ad-27 |
| 模块名称 | L4 长期层经验抽象提炼单元 |
| 所属分区 | 三、漏斗二：自动驾驶自成长漏斗 / 五层记忆层级存储 |
| 核心职责 | 从 L4 长期层已固化的多条同类成功经验中，提取通用驾驶规则、策略模式与技能范式。将具体场景相关的经验（如“雨天在中山路急刹”）转化为不依赖特定时间、地点、对象的抽象规则（如“雨天湿滑路面跟车时距须大于2.5秒”）。提炼结果反哺 L4 条目标记，并作为可脱敏共享的核心技能知识储备 |
| 依赖模块 | ad-26（L4 长期层存储单元，提供经验数据与接收提炼结果）、ad-36（综合重要度 I 值聚合计算单元，提供条目 I 值参考） |
| 被依赖模块 | ad-26（接收提炼出的通用规则与条目标记更新）、ad-50（记忆导入导出与脱敏共享单元，消费提炼出的泛化技能包） |


## 内部状态定义

| 状态 | 标识 | 含义 | 触发条件 |
|------|------|------|----------|
| 空闲等待 | `IDLE` | 无提炼任务，等待触发 | 漏斗二激活，L4 已初始化 |
| 数据拉取中 | `FETCHING` | 正在从 L4 批量拉取待提炼的经验条目 | 收到提炼请求或周期性触发 |
| 聚类分析中 | `CLUSTERING` | 正在按场景特征向量对经验进行分簇 | 数据拉取完成 |
| 规则归纳中 | `INDUCTING` | 正在从各簇经验中归纳通用规则 | 聚类完成 |
| 规则校验中 | `VERIFYING` | 正在校验新规则与已有知识库的冲突 | 规则归纳完成 |
| 结果回写中 | `WRITING_BACK` | 正在将提炼出的规则回写 L4 或标记源条目 | 校验通过 |
| 暂停服务 | `PAUSED` | L4 处于维护或冻结状态，暂停提炼 | 驾驶模式切换、L4 维护中 |

## 提炼规则引擎

### 一、聚类分析

**输入**：L4 中 `已提炼标记=False` 的成功经验（结果分类标签=“成功优化”）  
**方法**：基于场景特征向量的 DBSCAN 聚类  
**参数**：
- 最小簇条目数：3（默认，各分槽可独立配置）
- 聚类相似度阈值：0.75（默认，特殊环境槽 0.80）

**聚类特征维度**：
- 道路属性（等级、路面类型、车道数）
- 环境条件（天气、光照、时段）
- 交通参与情况（核心风险目标类别、交通流密度）
- 行为类型（跟车/变道/制动/转弯/让行等）

### 二、规则归纳

**从每个簇中归纳通用规则的方法**：
1. **提取共性约束**：在簇内所有条目共有的前置条件中，保留 80% 以上条目满足的条件
2. **泛化具体参数**：将具体的数值（如“跟车时距 2.3s”）归纳为分级建议（如“跟车时距 ≥ 2.5s”）
3. **剔除个性数据**：去除 GPS 坐标、精确时间戳、特定车牌/行人特征
4. **生成 IF-THEN 规则**：IF [泛化前置条件] THEN [建议驾驶策略]

**示例**：
- 源经验 1：雨天 + 高速 + 车速 100km/h，跟车时距 2.3s → 安全通过
- 源经验 2：雨天 + 高速 + 车速 90km/h，跟车时距 2.5s → 安全通过
- 源经验 3：雨天 + 高速 + 车速 110km/h，跟车时距 2.1s → 轻微惊险（I 值略低但成功）

→ **提炼规则**：IF 路面湿滑 AND 高速行驶 THEN 建议跟车时距 ≥ 2.5s，车速 ≤ 100km/h。置信度：0.85

### 三、冲突检测

| 冲突类型 | 检测方式 | 处理 |
|----------|----------|------|
| 新规则与已有规则矛盾 | 新规则的 THEN 建议与已有规则在相同 IF 条件下冲突 | 比较双方置信度与贡献条目数，保留更高者，记录冲突日志 |
| 新规则覆盖已有规则 | 新规则的 IF 条件是已有规则的子集，但 THEN 建议相同 | 合并，扩大适用范围 |
| 新规则过于宽泛 | 规则置信度 < 阈值（默认 0.60） | 不生成正式规则，仅保留为观察项 |


## 核心处理逻辑

```
FUNCTION l4_abstraction_main_loop():
    SET internal_state = IDLE
    提炼间隔 = 30 日
    最小 L4 条目数 = 10

    WHILE 系统运行中:
        // 第 1 步：全局调度
        IF 收到 ad-03 冻结指令 OR L4 处于维护状态:
            SET internal_state = PAUSED
            CONTINUE
        ELSE IF 激活指令 AND internal_state == PAUSED:
            SET internal_state = IDLE

        // 第 2 步：检查触发条件
        IF 收到 ad-26 提炼触发请求:
            GOTO 执行提炼
        ELSE IF internal_state == IDLE AND 距上次提炼 >= 提炼间隔:
            // 主动检查
            向 ad-26 查询当前 L4 条目总数
            IF L4 条目总数 > 最小 L4 条目数:
                GOTO 执行提炼

        SLEEP 60 秒
        CONTINUE

        LABEL 执行提炼:
        start_time = NOW()
        SET internal_state = FETCHING

        // 第 3 步：拉取待提炼经验
        向 ad-26 请求 L4 经验数据批量包（筛选条件：结果分类标签="成功优化" AND 已提炼标记=False）
        WAIT ad-26 返回批量包

        IF 批量包.条目数 < 最小 L4 条目数:
            SET internal_state = IDLE
            CONTINUE

        // 第 4 步：聚类分析
        SET internal_state = CLUSTERING
        场景特征向量列表 = 提取所有条目的场景特征向量
        聚类结果 = DBSCAN(场景特征向量列表, eps=聚类相似度阈值, min_samples=最小簇条目数)
        // 聚类结果：{簇编号 : [条目 ID 列表]}

        IF 聚类结果.簇数为 0:
            SET internal_state = IDLE
            CONTINUE  // 没有足够相似的条目簇

        // 第 5 步：逐簇归纳规则
        SET internal_state = INDUCTING
        新规则列表 = []
        条目标记更新列表 = []

        FOR EACH 簇 IN 聚类结果:
            簇条目列表 = 根据条目 ID 获取完整经验内容
            IF 簇条目列表.length < 最小簇条目数:
                CONTINUE

            // 5a. 提取共性前置条件
            共性 IF 条件 = 提取共性前置(簇条目列表, 满足率阈值=0.80)

            // 5b. 归纳驾驶策略
            建议 THEN 策略 = 归纳策略(簇条目列表)

            // 5c. 计算规则置信度
            规则置信度 = 计算置信度(簇条目列表, 共性 IF 条件, 建议 THEN 策略)

            // 5d. 生成规则
            新规则 = {
                规则 ID: 生成唯一规则 ID(),
                规则内容: "IF " + 共性 IF 条件 + " THEN " + 建议 THEN 策略,
                适用场景标签: 簇条目列表.来源分槽号 + 簇条目列表.子类标记,
                置信度: 规则置信度,
                贡献条目 ID 列表: 簇条目列表.各条目 ID
            }

            IF 规则置信度 >= 规则置信度阈值（默认 0.60）:
                新规则列表.append(新规则)
                条目标记更新列表.extend(簇条目列表.各条目 ID → {已提炼标记=True, 关联规则 ID=新规则.规则 ID})
            ELSE:
                // 置信度不足，仅标记不生成规则
                条目标记更新列表.extend(簇条目列表.各条目 ID → {已提炼标记=True, 关联规则 ID=NULL})

        // 第 6 步：冲突检测与合并
        SET internal_state = VERIFYING
        已有规则库 = 读取 L4 本地规则索引表
        最终新增规则 = []
        冲突规则列表 = []

        FOR EACH 新规则 IN 新规则列表:
            冲突类型 = 检测冲突(新规则, 已有规则库)
            IF 冲突类型 == "矛盾":
                // 比较置信度与贡献条目数
                已有规则 = 已有规则库[冲突规则 ID]
                IF 新规则.置信度 > 已有规则.置信度 AND 新规则.贡献条目数 > 已有规则.贡献条目数:
                    最终新增规则.append(新规则)
                    冲突规则列表.append({类型: "替换", 旧规则 ID: 已有规则.ID, 新规则 ID: 新规则.ID})
                ELSE:
                    冲突规则列表.append({类型: "保留旧", 旧规则 ID: 已有规则.ID, 新规则 ID: 新规则.ID})
            ELSE IF 冲突类型 == "覆盖":
                合并规则 = 合并(新规则, 已有规则)
                最终新增规则.append(合并规则)
                冲突规则列表.append({类型: "合并", 规则 ID: 合并规则.ID})
            ELSE:  // 无冲突
                最终新增规则.append(新规则)

        // 第 7 步：回写 L4 并记录日志
        SET internal_state = WRITING_BACK
        IF 最终新增规则不为空:
            向 ad-26 发送新生成规则列表（追加到 L4 规则索引表）
        IF 条目标记更新列表不为空:
            向 ad-26 发送条目标记更新指令（更新源条目的已提炼标记与关联规则 ID）
        IF 冲突规则列表不为空:
            向 ad-26、ad-03 发送冲突规则告警

        提炼结果报告 = {
            本轮提炼出规则数: 最终新增规则.length,
            更新规则数: 冲突规则列表中类型为“合并”和“替换”的数量,
            合并规则数: 冲突规则列表中类型为“合并”的数量,
            剔除规则数: 冲突规则列表中类型为“保留旧”的数量,
            提炼前后统计对比: {
                提炼前规则总数: 已有规则库.length,
                提炼后规则总数: 已有规则库.length + 最终新增规则.length - 替换数
            },
            提炼耗时: NOW() - start_time
        }
        向 ad-26 发送提炼结果报告
        向 ad-51 写入提炼事件日志（含完整提炼结果报告 + 所有新规则 + 冲突详情）

        // 第 8 步：状态上报
        向 ad-03、ad-01 上报提炼状态

        SET internal_state = IDLE
```


## 约束与异常处理

| 场景 | 处理方式 | 恢复条件 |
|------|----------|----------|
| 提炼过程中 L4 条目被其他操作修改或晋升 | 提炼使用数据快照，提炼完成后回写时若发现条目已不存在则跳过该条目标记 | — |
| 某簇条目共性 IF 条件几乎为空（无法归纳） | 该簇跳过，不生成规则，条目仍标记为“已提炼”防止重复尝试 | — |
| 新规则置信度始终无法达到阈值 | 不生成规则，仅标记条目为“已提炼”，下个周期可结合新条目重新聚类 | — |
| 提炼计算耗时超过 10 分钟 | 分批处理，每批处理 2 个簇，剩余簇暂停待下一周期继续 | 下一提炼周期 |
| 规则库条目数超过上限（> 1000 条） | 触发规则合并维护，合并置信度 ≥ 0.90 的相似规则 | 合并完成 |
| 冻结状态收到提炼请求 | 不处理，状态 PAUSED | 激活指令 |

## 安全边界

| 规则编号 | 内容 |
|:---:|------|
| S-01 | 提炼生成的规则不得包含任何原始场景的具体数据（GPS、时间戳、车牌、人脸特征等），仅保留结构化 IF-THEN 条件与策略 |
| S-02 | 新规则在正式存入 L4 规则索引表前，必须通过冲突检测，防止覆盖安全相关的高置信度已有规则 |
| S-03 | 策略失误经验（结果标签=“策略失误”）绝对不可参与规则提炼，仅成功经验可贡献规则 |
| S-04 | 不可抗力场景经验（结果标签=“不可抗力”）不参与常规提炼，但可单独聚类生成最高安全级别的警示规则（特殊处理） |
| S-05 | 提炼出的规则置信度低于 0.60 不可成为正式规则，仅保留为观察项，不向 ECC 大脑提供查询 |
| S-06 | 所有提炼操作（含生成、更新、合并、剔除）全量写入 ad-51 不可变日志 |


## 接口校验用例

| 用例编号 | 前置条件 | 输入 | 预期输出 |
|----------|----------|------|----------|
| TC-27-01 | IDLE，L4 有 15 条成功经验，其中 10 条高度相似（雨天+高速+跟车） | 提炼请求 | 形成 1 簇 → 归纳出 1 条 IF 湿滑路面+高速 THEN 跟车时距≥2.5s 的规则，置信度 ≥ 0.60，回写 L4 |
| TC-27-02 | IDLE，L4 有 12 条成功经验，但无足够相似条目 | 提炼请求 | 聚类无结果，不上报规则，标记所有条目为“已提炼” |
| TC-27-03 | IDLE，新规则与已有规则冲突（矛盾），新规则置信度更高 | 新规则置信度 0.90 vs 旧规则 0.70 | 替换旧规则，记录冲突日志 |
| TC-27-04 | IDLE，新规则置信度 0.50（低于阈值） | 新规则置信度 0.50 | 不生成正式规则，条目仅标记“已提炼” |
| TC-27-05 | IDLE，提炼数据中包含 1 条策略失误经验 | 经验数据批量包 | 策略失误经验被筛选过滤，不参与聚类 |
| TC-27-06 | WRITING_BACK，回写时发现某条目已晋升至 L5 | 条目标记更新指令 | 该条目标记跳过，其他正常回写 |
| TC-27-07 | 提炼耗时超过 10 分钟 | 提炼还在进行中 | 暂停当前批次，剩余簇下一周期继续 |
| TC-27-08 | 规则库已有 1000 条规则 | 新规则生成后 | 触发规则合并维护，合并相似规则 |
| TC-27-09 | 提炼成功 | 提炼结果报告 | 报告含本轮提炼出规则数、冲突处理详情，写入 ad-51 |
| TC-27-10 | PAUSED | 提炼请求 | 不处理，等待激活 |
