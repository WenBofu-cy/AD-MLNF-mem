# ad-37-重要度增量定时刷新单元 接口规格

---

## 基本信息

| 项 | 内容 |
|----|------|
| 模块编号 | ad-37 |
| 模块名称 | 重要度增量定时刷新单元 |
| 所属分区 | 三、漏斗二：自动驾驶自成长漏斗 / 三维重要度计算引擎 |
| 核心职责 | 周期性地对漏斗二中所有经验条目的重要度 I 值执行增量刷新。驱动 S 值、V 值、C 值随时间与环境变化而更新的因子重新参与 I 值聚合计算，确保所有条目的重要度始终保持对当前驾驶环境的时效性。避免长期未被主动更新的条目因 I 值失真而影响晋升或遗忘判定的准确性 |
| 依赖模块 | ad-36（综合重要度 I 值聚合计算单元，接收重算信号）、ad-31（S 值计算单元，可请求回溯更新）、ad-32（V 值计算单元，可请求回溯更新）、ad-33（C 值统计单元，可请求回溯更新） |
| 被依赖模块 | ad-36（消费批量重算触发信号） |


## 内部状态定义

| 状态 | 标识 | 含义 | 触发条件 |
|------|------|------|----------|
| 空闲等待 | `IDLE` | 等待下一个刷新周期 | 系统初始化完成，或上次刷新结束 |
| 常规刷新中 | `REFRESHING_NORMAL` | 正在执行每日定时刷新 | 距上次常规刷新 ≥ 24 小时 |
| 紧急刷新中 | `REFRESHING_URGENT` | 因 S 值历史基准大幅变化、环境剧变或权重变更触发的紧急刷新 | 外部剧变信号或手动触发 |
| 暂停服务 | `PAUSED` | 系统紧急熔断或维护，暂停刷新 | 收到 ad-01 紧急熔断指令或驾驶模式切换 |


## 输入数据

| 输入项 | 数据类型 | 来源模块 | 触发条件 | 优先级 |
|--------|----------|----------|----------|:---:|
| 系统时钟信号 | UTC 时间戳 | 系统时钟 | 周期性，每 60 秒 | 普通 |
| 外部剧变信号 | Struct（剧变类型：季节变化/长期施工结束/权重变更/环境突变） | ad-01 总控漏斗 F₀、ad-35 | 检测到可能使大量经验 S/V 值失真的环境或配置变化 | **高** |
| 手动刷新指令 | Signal | ad-01（管理员触发） | 人工运维或系统恢复后 | **高** |
| 全局调度指令 | Enum（冻结/激活） | ad-03 漏斗二专属调度单元 | 模式切换时 | **高** |
| 当前全量条目列表 | List（条目 ID + 上次 I 值更新时间） | ad-36 综合重要度 I 值聚合计算单元 | 刷新开始时请求 | **高** |


## 输出数据

| 输出项 | 数据类型 | 目标模块 | 输出条件 | 优先级 |
|--------|----------|----------|----------|:---:|
| 常规批量重算触发信号 | Signal（重算范围：全量 I 值更新时间 > 24h 的条目） | ad-36 综合重要度 I 值聚合计算单元 | 常规刷新触发时 | **高** |
| 紧急批量重算触发信号 | Signal（重算范围：全量条目 或 指定条目 ID 列表） | ad-36 | 紧急刷新触发时 | **高** |
| 因子回溯更新请求 | Struct（条目 ID + 请求更新因子类型：S / V） | ad-31、ad-32 | 刷新发现某条目因子陈旧时 | 普通 |
| 刷新结果汇总 | Struct（刷新类型 + 刷新条目总数 + 更新条目数 + I 值显著变化条目数 + 耗时 + 异常条目清单） | ad-03、ad-01、ad-51 | 每次刷新完成后 | 普通 |
| 刷新状态上报 | Struct（当前状态 + 距下次刷新时间 + 上次刷新结果摘要） | ad-03、ad-01 | 每 10 分钟周期性上报 | 普通 |


## 刷新策略与判定规则

### 一、常规刷新（每 24 小时）

**触发条件**：距上次常规刷新完成 ≥ 24 小时，且系统处于自动驾驶模式（漏斗二激活）。

**刷新范围**：所有在 I 值缓存中 `上次 I 值更新时间 > 24 小时` 的条目。

**刷新方式**：向 ad-36 发送常规批量重算触发信号。由 ad-36 使用缓存中的已有因子（I₀, S, V, C）重新计算 I 值。若某因子（如 S, V）超过 7 日未更新，则向对应模块请求回溯更新，使用当前环境参数重新评估。

### 二、紧急刷新

**触发条件**：收到外部剧变信号或手动刷新指令。

**剧变类型与刷新范围**：

| 剧变类型 | 刷新范围 | 理由 |
|----------|----------|------|
| 权重系数变更（ad-35） | 全量条目 | 权重直接影响 I 值 |
| 季节剧变（冬季→夏季） | 含冰雪/低温特殊环境经验的条目 | 环境因子基准改变 |
| 长期施工结束 | 含施工区场景标签的条目 | 该场景不再存在 |
| 用户驾驶风格切换 | 全量条目 | V 值基准改变 |

**刷新方式**：向 ad-36 发送紧急批量重算触发信号（指定范围）。同时请求 ad-31（S 值）、ad-32（V 值）使用当前最新环境参数对指定条目进行回溯重算，再由 ad-36 聚合。

### 三、异常监控与触发

在常规刷新间隙，若监测到以下情况，自动提前触发局部刷新：
- L3/L4 层条目晋升成功率显著下降（> 30% 条目多次晋升失败）
- ad-40 遗忘候选清单中 I 值异常条目占比 > 20%
- ad-31 报告全局安全信号基准偏移 > 30%


## 核心处理逻辑

```
FUNCTION i_refresh_scheduler_main_loop():
    SET internal_state = IDLE
    last_normal_refresh_time = NOW()
    常规刷新间隔 = 24 小时
    因子陈旧阈值 = 7 日

    WHILE 系统运行中:
        // 第 1 步：全局调度
        IF 收到 ad-03 冻结指令:
            SET internal_state = PAUSED
            CONTINUE
        ELSE IF 激活指令 AND internal_state == PAUSED:
            SET internal_state = IDLE
            CONTINUE

        // 第 2 步：最高优先级——紧急刷新
        IF 收到外部剧变信号 OR 手动刷新指令:
            SET internal_state = REFRESHING_URGENT
            start_time = NOW()
            刷新范围 = 根据剧变类型确定条目范围

            // 2a. 请求相关因子回溯更新
            IF 剧变类型 IN [权重变更, 风格切换, 季节剧变]:
                向 ad-31 发送回溯更新请求(刷新范围)
                向 ad-32 发送回溯更新请求(刷新范围)
                WAIT ad-31、ad-32 返回更新完成（最长等待 60 秒）

            // 2b. 触发 I 值聚合重算
            向 ad-36 发送紧急批量重算触发信号(刷新范围)
            WAIT ad-36 返回重算完成

            向 ad-03、ad-01、ad-51 发送刷新结果汇总
            last_normal_refresh_time = NOW()  // 紧急刷新重置常规计时器
            SET internal_state = IDLE
            CONTINUE

        // 第 3 步：常规刷新检查
        IF internal_state == IDLE AND NOW() - last_normal_refresh_time >= 常规刷新间隔:
            // 仅在自动驾驶模式下执行常规刷新
            IF 驾驶模式 == 自动驾驶:
                SET internal_state = REFRESHING_NORMAL
                start_time = NOW()

                // 3a. 获取全量条目列表
                向 ad-36 请求当前全量条目列表
                WAIT 返回列表
                陈旧条目列表 = 筛选(列表, 条目.上次I值更新时间 > 24h)

                IF 陈旧条目列表不为空:
                    // 3b. 检查因子时效性
                    超陈旧条目列表 = 筛选(陈旧条目列表, 条目.上次S或V更新时间 > 7日)
                    IF 超陈旧条目列表不为空:
                        向 ad-31 发送回溯更新请求(超陈旧条目列表)
                        向 ad-32 发送回溯更新请求(超陈旧条目列表)
                        WAIT 返回更新完成

                    // 3c. 触发 I 值批量重算
                    向 ad-36 发送常规批量重算触发信号(陈旧条目列表)
                    WAIT ad-36 返回重算完成

                向 ad-03、ad-01、ad-51 发送刷新结果汇总
                last_normal_refresh_time = NOW()
                SET internal_state = IDLE
            CONTINUE

        // 第 4 步：异常监控（在空闲时低频率运行）
        IF internal_state == IDLE AND 距上次异常监控 >= 30 分钟:
            向 ad-38 查询近期晋升成功率
            向 ad-40 查询遗忘异常占比
            IF 晋升成功率下降 > 30% OR 遗忘异常占比 > 20%:
                向 ad-03、ad-01 上报告警（建议手动触发紧急刷新）

        // 第 5 步：周期性状态上报
        IF 距上次上报 >= 10 分钟:
            向 ad-03、ad-01 上报刷新状态

        SLEEP 60 秒
```


## 约束与异常处理

| 场景 | 处理方式 | 恢复条件 |
|------|----------|----------|
| 常规刷新期间系统切换为非自动驾驶 | 暂停常规刷新，保留进度，下次进入自动驾驶后从断点继续 | 驾驶模式恢复自动驾驶 |
| 刷新请求 ad-36 超时无响应（> 5 分钟） | 上报告警，标记本次刷新失败，等待下一个周期重试 | ad-36 恢复正常 |
| 紧急刷新期间收到另一紧急刷新请求 | 加入刷新队列，当前刷新完成后依次执行 | 队列清空 |
| 全量条目数量极大（> 50,000） | 分批刷新，每批 5,000 条，批次间释放资源 | 刷新完成 |
| 手动刷新指令但令牌无效 | 拒绝执行，上报告警 | — |
| 系统时间被异常调整导致刷新计时器错乱 | 使用系统时钟单调递增计时，不受绝对时间跳变影响 | — |

## 安全边界

| 规则编号 | 内容 |
|:---:|------|
| S-01 | 常规刷新间隔 24 小时为编译期默认值，可在 [12h, 72h] 范围内调整，不得低于 12 小时以防过度刷新消耗系统资源 |
| S-02 | 紧急刷新仅能由 ad-01 授权的剧变信号或管理员手动触发，不可由用户级指令触发 |
| S-03 | 刷新过程为只读分析+批量计算，不得直接修改任何条目的原始经验数据、时间戳或锁定状态 |
| S-04 | L5 核心层条目的 I 值参与刷新计算（因 C 值、权重可能变化），但原始经验数据与锁定状态不受任何影响 |
| S-05 | 刷新过程中若发生紧急安全事件，立即中断刷新，优先响应安全信号 |
| S-06 | 所有刷新操作（含常规、紧急、失败）全量写入 ad-51 不可变日志 |


## 接口校验用例

| 用例编号 | 前置条件 | 输入 | 预期输出 |
|----------|----------|------|----------|
| TC-37-01 | IDLE，距上次刷新 25 小时，自动驾驶模式 | 系统时钟触发 | 启动常规刷新，筛选陈旧条目，请求回溯更新超陈旧因子，触发 ad-36 重算 |
| TC-37-02 | IDLE，收到权重变更剧变信号 | 剧变信号（类型=权重变更） | 启动紧急刷新，全量条目回溯更新 S/V，触发 ad-36 全量重算 |
| TC-37-03 | REFRESHING_NORMAL，驾驶模式切换为人工驾驶 | 驾驶模式变更 | 暂停刷新，保留进度，下次恢复后从断点继续 |
| TC-37-04 | IDLE，收到手动刷新指令（有效令牌） | 手动刷新指令 | 启动紧急刷新，范围=全量 |
| TC-37-05 | 全量条目 60,000 条 | 常规刷新触发 | 分 12 批执行，每批 5,000 条，批次间释放资源 |
| TC-37-06 | 刷新中 ad-36 超时 6 分钟 | 重算请求无响应 | 标记刷新失败，上报告警，等待下一周期 |
| TC-37-07 | IDLE，晋升成功率下降 35% | 异常监控触发 | 上报告警，建议手动触发紧急刷新 |
| TC-37-08 | 紧急刷新中收到另一紧急刷新请求 | 第二次紧急刷新请求 | 加入刷新队列，当前刷新完成后依次执行 |
| TC-37-09 | PAUSED（紧急熔断） | 系统时钟触发常规刷新 | 不处理 |
| TC-37-10 | 刷新完成 | ad-36 返回完成 | 汇总结果，上报 ad-03、ad-01、ad-51，更新上次刷新时间 |
